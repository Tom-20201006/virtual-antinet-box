# Commands

`docs/01_COMMANDS.md` 是新结构下的主命令入口。Codex 桌面端每轮只执行用户指定的一个 CMD 编号命令，不自行选择其他命令。

## 状态枚举

- `PENDING`：等待执行。
- `IN_PROGRESS`：正在执行。
- `DONE`：已完成并记录结果。
- `BLOCKED`：被外部条件阻塞，需要用户处理或确认。
- `FAILED`：执行失败。
- `CANCELLED`：已取消。

### 任务目标

根据当前项目目标，将 Godot 项目中的桌面、纸质卡片、卡片盒从“可运行的功能性 Demo”重构为更接近现实世界物体的实体组件系统。

本轮任务不以 Antinet 的抽象功能分类为中心，也不以知识管理流程为中心，而是以现实物体的几何、材质、厚度、约束、运动方式、空间关系和可交互反馈为中心。

重点对象：

1. 桌面：作为连续的现实工作面，而不是软件分区界面。
2. 纸质卡片：作为有尺寸、厚度、正反面、边缘、纸面纹理和墨迹层的物体，而不是按 Main Card / Bibcard / Index Card 等功能分类的 UI 卡片。
3. 卡片盒 / notebox：按照卢曼论文中的物理描述重构为木质盒体、可拉开的抽屉、抽屉内部卡片序列、普通纸片和可见固定位置。

本轮目标是让虚拟空间首先像一个真实桌面上的实体卡片盒系统，再让 Antinet 语义在后续阶段附着到这些实体物品上。

### 设计哲学

本轮必须遵守以下设计哲学：

1. 先复原物体，再承载意义。
2. 不要把卡片先分成 Main Card、Bibcard、Index Card 等功能性类别。
3. 卡片首先是纸片对象，其差异来自物理属性：尺寸、厚度、颜色、材质、横线、空白、正反面、边缘、书写痕迹。
4. 桌面是连续工作面，不是分区 UI。不要在桌面上创建“阅读区、写作区、待归档区、索引区”这类软件式固定分区。
5. 抽屉是可拉出的物理容器，不是文件夹列表。
6. 抽屉内的卡片是一个有顺序、有密度、有缝隙、有前后邻接关系的物理队列。
7. 插入卡片应表现为进入两张卡之间的物理缝隙，而不是单纯调用“保存到地址”。
8. 地址、链接、引用等暂时只应表现为纸面上的可见标记或占位区域，不要发展成搜索、标签、OCR、自动分类、AI 摘要或知识图谱功能。
9. 允许使用 Godot 的近似物理、约束运动、动画和交互辅助来模拟现实手感；不要试图实现高成本的真实纸张有限元、复杂软体纸堆或完全真实摩擦仿真。
10. 本轮重构应以可运行、可验收、可继续扩展为目标，而不是追求一次性完成真实世界全部物理细节。

### 理论依据

本轮对卡片盒的基础描述必须以卢曼论文《Communication with Noteboxes》中的技术要求为依据：

- 使用木盒。
- 木盒有可拉开的抽屉。
- 使用约 octavo 格式的纸片，约为 half letter-size sheet。
- 纸片只在一面书写，这样翻找时不必把纸片取出就能阅读。
- 卢曼建议使用普通纸而不是厚卡纸，因为几十年后系统会变得很大，空间占用会影响使用。
- 每张纸片有固定位置；编号应可见且不应改变。
- 系统依赖固定位置、局部插入和向内生长，而不是依赖预先的整体主题分类。

本轮也可以参考 Scheper 对 A6 / 4x6 英寸卡片、空白双面卡、彩色 ruled index cards、字母分隔片、不同颜色笔迹的描述，但这些只作为物理差异参考，不作为功能分类依据。

#### 2. 桌面重构

将桌面重构为一个连续的现实工作面：

- 桌面应是一个足够大的实体平面。
- 有木质或纸质工作台质感，允许简化材质。
- 桌面不划分固定功能区。
- 桌面上的物体可以自由放置、重叠、旋转、靠近、分散。
- 桌面应保存物体的空间位置、旋转角度、上下叠放关系或至少保存平面位置与朝向。
- 桌面视觉上应像现实工作台，而不是软件 dashboard。

如果当前已有导入区、摊开区、卡片盒区等视觉分区，应弱化或移除这些固定分区，只保留自然摆放关系。

#### 3. 纸质卡片重构

将卡片重构为物理纸片对象，而不是功能性 UI 卡片。

每张卡片至少应具备：

- 长宽比例：默认接近 4x6 英寸或 A6，允许在项目配置中表达比例。
- 厚度：必须可见，不能只是无厚度平面。
- 正面。
- 背面。
- 边缘。
- 纸张材质。
- 轻微纸张颜色差异。
- 可选横线纹理或空白纹理。
- 可见地址/编号占位区域，但不要实现自动编号逻辑。
- 可见墨迹层或 placeholder 书写痕迹。
- 可被拾取、拖动、放下、翻面、旋转。

卡片差异只按物理属性表达，例如：

- 空白纸卡。
- 有横线纸卡。
- 彩色纸卡。
- 薄普通纸。
- 略厚索引卡。
- 较高分隔片。
- 带顶部标签的分隔片。

不要按 Antinet 语义功能建立不同类，例如不要建立 `MainCardNode`、`BibCardNode`、`IndexCardNode` 这种语义类型作为本轮核心。可使用通用 `PaperCard` / `CardPhysicalObject` 之类基类，通过物理参数变化表达差异。

#### 4. 卡片盒 / notebox 重构

按照卢曼论文中的描述，将卡片盒重构为现实 notebox：

- 木质盒体。
- 可拉开的抽屉。
- 抽屉带前面板。
- 抽屉带把手。
- 抽屉可沿单一轴线滑动。
- 抽屉有关闭、半开、全开状态。
- 抽屉打开后露出内部卡片队列。
- 抽屉内部应表现为一个长方体容器，包含底板、侧壁、前挡板、后挡板或等价简化结构。
- 抽屉外侧可有标签槽或铭牌，但不要把标签槽做成软件目录。

抽屉运动应使用 Godot 可控动画或物理约束实现，不要求完全真实滑轨仿真，但必须看起来像现实抽屉沿导轨拉出。

#### 5. 抽屉滑轨和开合反馈

实现或模拟滑轨约束：

- 抽屉只能沿一个方向拉出和推回。
- 鼠标拖动或点击交互应让抽屉沿轴线移动。
- 抽屉到达最大拉出距离时停止。
- 抽屉关闭时卡片队列应被盒体部分遮蔽。
- 抽屉打开时内部卡片队列逐渐可见。

可以采用 kinematic movement、Tween、AnimationPlayer、约束脚本或 Godot 物理节点组合实现。不要为了真实物理牺牲交互稳定性。

#### 6. 抽屉内卡片队列

抽屉内卡片应以物理队列方式呈现：

- 卡片按前后顺序站立或略倾斜排列。
- 卡片之间有极小间距。
- 多张卡形成可见厚度和密度。
- 可看到卡片上边缘或侧边缘。
- 分隔片应比普通卡略高，顶部标签能从队列中露出。
- 用户可以从队列中选择某一张卡。
- 用户可以把卡片从队列中抽出到桌面。
- 抽出后应保留队列中的空位感或让附近卡片轻微闭合。
- 插入卡片时应显示两张卡之间的缝隙或高亮插入位置。

本轮不必实现复杂大规模卡片性能优化，但至少要构造一个小规模可验收样例，体现真实队列关系。

#### 7. 分隔片重构

分隔片按照物理属性处理，不按知识功能处理：

- 分隔片是更高的纸片。
- 分隔片顶部有突出的 tab。
- tab 上可有字母或短标签 placeholder。
- 分隔片与普通卡在同一队列中排列。
- 分隔片用于视觉定位，但本轮不要做成树状目录或标签系统。

#### 8. 纸面标记与墨迹

在卡片上建立视觉标记层：

- 地址占位标记。
- 普通黑色书写痕迹 placeholder。
- 绿色链接标记 placeholder。
- 红色外部引用标记 placeholder。
- 线条、划线、箭头等纸面痕迹。

这些标记在本轮只作为纸面视觉属性，不作为可搜索数据结构。

#### 9. 交互方式

本轮交互重点是直接操纵物体：

- 点击或拖动抽屉把手打开抽屉。
- 点击卡片选中。
- 拖动卡片到桌面。
- 拖动桌面卡片改变位置。
- 旋转卡片。
- 翻面卡片。
- 将卡片拖回抽屉中的某个物理缝隙。

不要把核心交互做成菜单驱动、表单驱动、列表驱动或搜索驱动。

#### 10. 保存状态

如果当前项目已有保存/加载机制，应更新它以至少保存：

- 卡片是否在桌面上或抽屉内。
- 卡片在桌面上的位置和旋转。
- 卡片正面/背面朝向。
- 抽屉打开程度。
- 卡片在抽屉队列中的顺序。
- 分隔片在队列中的位置。
- 卡片物理类型参数：大小、厚度、颜色、是否有横线、是否为较高分隔片。

不要保存 OCR、标签、自动分类结果或 AI 生成数据。

#### 11. 视觉验收样例

本轮完成后，主场景中至少应能看到：

- 一个连续桌面。
- 一个木质卡片盒或 notebox。
- 至少一个可拉开的抽屉。
- 抽屉中一组有厚度的纸质卡片。
- 至少一种普通纸卡。
- 至少一种横线或彩色纸卡。
- 至少一个较高的分隔片。
- 卡片能被从抽屉中拿到桌面。
- 桌面上的卡片可移动、旋转、翻面。
- 卡片能插回抽屉队列中的物理位置。

### 实现策略建议

Codex 可根据项目现状自行选择实现方式，但应优先保证稳定性。

推荐策略：

1. 用 MeshInstance3D / CSG / simple primitive 先建立清晰几何形体。
2. 用材质和贴图 placeholder 表达木头、纸张、墨迹。
3. 用 CharacterBody3D、RigidBody3D、StaticBody3D、Area3D、RayCast、Tween 或 AnimationPlayer 组合实现交互。
4. 对卡片使用薄长方体 mesh，而不是单面 plane。
5. 对抽屉使用脚本约束的 kinematic 滑动，而不是完全依赖自由刚体物理。
6. 对卡片队列使用可控的排列和吸附，而不是完全真实纸堆碰撞。
7. 对插入使用可见缝隙、高亮、吸附目标点来模拟现实插卡。

如果项目当前架构不适合一次性重构全部内容，可以先建立一个独立 `PhysicalPrototype` 场景或 equivalent demo scene，但主入口必须能进入或展示该新物理复原结果。
