# Decisions

## DEC-2026-06-18-001 使用 GitHub 作为事实来源

决策：GitHub 仓库是项目唯一事实来源。

理由：iOS ChatGPT、Codex 桌面端和用户远程验收需要共享同一套可追溯状态，不能依赖单一聊天窗口。

## DEC-2026-06-18-002 iOS ChatGPT 直接写命令

决策：iOS ChatGPT 负责与用户讨论，并把编号化命令写入仓库命令文档。

理由：减少手机端远程桌面的长文本输入负担。

## DEC-2026-06-18-003 Codex 只执行指定 CMD

决策：Codex 桌面端每轮只执行用户指定的一个 CMD 编号命令。

理由：避免自动扩大任务范围，便于用户审查和回滚。

## DEC-2026-06-18-004 最低限度数字化边界

决策：本项目不做 OCR、搜索、标签、自动分类、自动链接、AI 摘要、知识图谱、Markdown 笔记编辑器、云同步、账号系统或任何基于卡片内容的智能处理。

理由：项目目标是模拟实体卡片盒的物理承载层，不替用户判断卡片内容。

## DEC-2026-06-18-005 验收步骤标准化

决策：后续所有验收步骤必须按 `docs/validation.md` 的逐步标准编写。

理由：避免只写“确认正常”，保证人工验收可复现、可观察、可记录。

## DEC-2026-06-19-001 文档按使用频率和职责分层

决策：新文档结构采用根目录核心文档、docs 主控层、spec 规格层、research 研究层、history 正式历史层、archive 冷归档层。

理由：降低每轮 Codex 和 iOS ChatGPT 的上下文读取成本，避免当前状态、历史、研究和规格混杂。

## DEC-2026-06-19-002 history 与 archive 分离

决策：history 保存正式历史和摘要，archive 保存旧文件原文、旧命令全文、旧报告和原始材料。

理由：正式历史应可读，冷归档应完整，二者职责不同。
